# FIPS 140-3

## Feynman Explanation
FIPS 140-3 is the *U.S. federal certification* that says: "this piece of software or hardware that does cryptography is built to a specified level of tamper resistance, key management hygiene, and self-testing, and a NIST-accredited lab has verified it." Vendors who want to sell to the U.S. government (and to most banks, healthcare networks, and large enterprises that follow federal standards) must put their crypto modules through CMVP validation. There are four levels — from "tried and true software on a server" (Level 1) to "hardened hardware with physical tamper evidence and zeroization on intrusion" (Level 4) — and the level you need depends on the value of the keys, the threat model, and the regulator's mandate.

## Technical Details

### 1. The Standard

FIPS 140-3 (2019) supersedes FIPS 140-2 (2001). It is the U.S. implementation of **ISO/IEC 19790:2012** *Security requirements for cryptographic modules*. The transition ended 21 September 2021; only FIPS 140-3 validations are accepted now.

**Companion standard**: ISO/IEC 24759:2017 — test requirements for the modules.

**Authoritative bodies**:
- **NIST** (National Institute of Standards and Technology) — standard owner, runs CMVP.
- **CSEC** (Communications Security Establishment Canada) — Canadian equivalent.
- **NVLAP** (National Voluntary Laboratory Accreditation Program) — accredits the test labs (CCTLs).
- **CCTL** (Cryptographic and Security Testing Laboratory) — performs the validation testing.

### 2. Cryptographic Module Validation Program (CMVP)

```
   Vendor ──→ CCTL (lab) ──→ NIST CMVP ──→ Validation Certificate
              testing          review
```

1. **Vendor** submits a *Security Policy* document, source code, and module to a CCTL.
2. **CCTL** runs the *Cryptographic Algorithm Validation Program (CAVP)* tests on each algorithm (AES, RSA, SHA, etc.) — algorithm-by-algorithm conformance.
3. **CCTL** runs the *Derived Test Requirements (DTR)* — module-level tests: physical security, key management, self-tests, EMI/EMC.
4. **NIST CMVP** reviews the lab's report; if successful, issues a *validation certificate* listed on the CMVP database.

The **CAVP** validates the algorithm correctness. The **CMVP** validates the *module* (key handling, interfaces, role-based authentication, physical security).

### 3. Module Definition

A **cryptographic module** is "the set of hardware, software, and/or firmware that implements approved cryptographic functions and is contained within a cryptographic module boundary." Examples:
- An HSM (Hardware Security Module).
- A TLS library's `crypto` subsystem.
- A smart card chip.
- An OpenSSL FIPS provider.

A module has:
- **Module boundary** — the logical or physical perimeter.
- **Cryptographic boundary** — subset of the module where crypto operations occur.
- **Ports and interfaces** — explicit data flow points.
- **Roles** — User, Crypto Officer (CO), Maintenance.
- **Services** — show status, show parameters, perform crypto.
- **Approved security functions** — algorithms on the FIPS-approved list.
- **Critical Security Parameters (CSPs)** — keys, key components, authentication data, intermediate values.
- **Public Security Parameters (PSPs)** — public keys, public parameters.

### 4. The Four Security Levels

#### 4.1 Level 1 — Software-only, Production-Grade

- **Physical security**: production-grade enclosure; **no** tamper evidence required.
- **Cryptographic algorithms**: must use FIPS-approved algorithms (AES, SHA, RSA, etc.) when invoked through the validated module boundary. Non-approved algorithms may be present but are flagged.
- **Self-tests**: power-up self-tests; conditional self-tests on demand.
- **Use case**: software crypto on a general-purpose server; TLS libraries in FIPS mode; the *default* level for most cloud services.
- **Examples**: OpenSSL 3.x FIPS provider, AWS LibCrypto (FIPS validated), Google BoringCrypto (FIPS validated).

#### 4.2 Level 2 — Tamper-Evident, Role-Based Authentication

Adds to L1:
- **Physical security**: tamper-evident seals, coatings, or pick-resistant locks. Opacity requirements.
- **Role-based authentication**: the Crypto Officer and User must authenticate (e.g., password, smart card).
- **Operational environment**: includes an EAL2+ evaluated OS (or similar) per [PP-CONFIG].
- **Use case**: software on a server in a *physically controlled* data center; module-level separation.
- **Examples**: most commercial HSMs at entry level; HSMs sold for payment processing.

#### 4.3 Level 3 — Tamper-Resistant, Identity-Based Authentication

Adds to L2:
- **Physical security**: *tamper-resistant* enclosure — the module must detect and respond to physical intrusion. Strong opacity. Zeroization on tamper detection.
- **Identity-based authentication**: per-user authentication (not just shared CO password); strong cryptographic identity.
- **Trusted path**: for entering CSPs (e.g., PIN entry over a secure channel).
- **EMI/EMC**: meets FCC Class B / equivalent.
- **Use case**: **HSMs that protect root keys**, code-signing keys, PIV credentials.
- **Examples**: Thales Luna PCIe, Entrust nShield, AWS CloudHSM (Level 3 FIPS 140-3 validated).

#### 4.4 Level 4 — Highest Physical Security

Adds to L3:
- **Physical security**: *tamper-detecting* enclosure that responds to *physical penetration* with zeroization, even if attempted from multiple directions.
- **EFT/Voltage protection**: the module survives and responds to environmental fault injection (voltage, temperature).
- **Use case**: military / Top Secret deployments, the absolute highest value keys.
- **Examples**: very few commercial offerings; bespoke military HSMs.

### 5. Self-Tests

Every FIPS 140-3 module must run:

#### 5.1 Power-Up Self-Tests (POST)
Run automatically before the module is operational:
- **Known-answer tests (KAT)** for every approved algorithm (encrypt "the answer is 42" → check the output).
- **Firmware integrity test** (HMAC or signature over the module's binary).
- **Critical function tests** for KDFs, RNG, DRBG seeding.

If any test fails, the module enters an *error state* and *zeroizes* all CSPs.

#### 5.2 Conditional Self-Tests
Run when triggered:
- **Pairwise consistency test** (PCT) for every generated key pair: sign with private, verify with public.
- **Continuous RNG test**: detect RNG failure in real time.
- **Bypass test**: if the module has a "bypass" mode, test that it is correct.

#### 5.3 Periodic Self-Tests
Re-run POST on operator demand.

### 6. Key Management Requirements

The module must implement:

1. **Key generation** — approved DRBG (e.g., AES-256-CTR_DRBG).
2. **Key establishment** — approved KAS (Key Agreement Scheme), e.g., SP 800-56A, SP 800-71.
3. **Key entry/output** — encrypted, with split-knowledge for > Level 2.
4. **Key storage** — CSPs encrypted at rest; zeroizable.
5. **Key zeroization** — overwrite the storage location with zeros/random; ideally voltage-driven.
6. **Key backup / escrow** — same protection as live key (split-knowledge, M-of-N).
7. **Key destruction** — physical destruction of media for archived keys (NIST SP 800-88).

### 7. Roles and Services

#### 7.1 Roles
- **User** — can use the module's crypto services.
- **Crypto Officer (CO)** — can configure, manage, install the module; can zeroize.
- **Maintenance** — service the module; must zeroize on entry.
- **Bypass** (optional) — services that bypass the cryptographic protection.

#### 7.2 Authentication
- Level 1: implicit (single user assumed).
- Level 2: role-based (password or token).
- Level 3: identity-based (per-user; strong crypto).
- Level 4: identity-based + multi-factor (or split-knowledge for shared secrets).

### 8. Approved Algorithms (SP 800-131A Rev. 2, 2023)

| Use | Approved |
|-----|----------|
| Symmetric encryption | AES-128, AES-192, AES-256 |
| Hashing | SHA-1 (legacy), SHA-224, SHA-256, SHA-384, SHA-512, SHA3-224, SHA3-256, SHA3-384, SHA3-512 |
| MAC | HMAC-SHA-1, HMAC-SHA-256, HMAC-SHA-384, HMAC-SHA-512, KMAC |
| RNG | Hash_DRBG, HMAC_DRBG, CTR_DRBG (SP 800-90A) |
| KEM / key exchange | KAS-SSC, KAS-FFC, KAS-ECC (SP 800-56A); ML-KEM-768 / 1024 (post-2024) |
| Signature | RSA-PSS, RSA-PKCS#1 v1.5 (legacy), ECDSA, EdDSA, ML-DSA, SLH-DSA |
| Key wrapping | AES-KW, AES-KWP, RSA-OAEP |

**Disallowed**: MD5, RC4, DES, 3DES (after 2023), RSA-1024, DSA, ECDSA with weak curves, static DH, ECB mode.

### 9. CMVP Validation Lifecycle

```
Module developed
  → Security Policy document written (module description, interfaces, services, CSPs)
  → CAVP algorithm testing (each algorithm in the module)
  → CCTL test report (DTR, physical security, key management, EMI/EMC)
  → CMVP review (typically 6–18 months)
  → Validation certificate issued
  → Module listed on https://csrc.nist.gov/projects/cryptographic-module-validation-program
  → Annual re-validation (maintenance) — vendor submits changes; CMVP issues updated certificate
```

The current CMVP backlog is *significant* (12–18 months for a new validation); vendors are encouraged to start early.

### 10. FIPS-140-3 vs FIPS 140-2 — Material Differences

| Aspect | FIPS 140-2 | FIPS 140-3 |
|--------|-------------|-------------|
| Year | 2001 | 2019 |
| Spec basis | Standalone | ISO/IEC 19790:2012 |
| Physical security | 4 levels | 4 levels (more detailed environmental testing) |
| Software security | Flaw remediation guidance | Lifecycle, development, deployment guidance |
| Sensitive security parameters | CSPs | CSPs + PSPs |
| Self-tests | Required | Required + tighter KAT/DRBG coverage |
| Non-invasive attacks | n/a | Mitigations required at higher levels |
| Module validation | n/a | Vendor affirmation in Security Policy |

In practice, FIPS 140-3 is *stricter* on the *software / lifecycle* side (development process, vulnerability handling) and *similar* on physical security.

### 11. The FIPS Boundary

A common pitfall: a *product* is FIPS-validated, but the *application* calls the wrong module. The boundary matters.

**Example**: OpenSSL 3.x has a *FIPS provider* (the validated module) and a *default provider* (not validated). Code that uses `EVP_aes_256_gcm()` (default) is *not* in FIPS mode; code that uses `EVP_FIPS_approved_cipher()` is.

Configuration:
- OpenSSL 3.x: load the FIPS provider, set the default to FIPS, ensure *all* algorithm invocations are FIPS-approved.
- HSM vendors: only the HSM client library is part of the validated boundary; the application calling the HSM is not in the boundary.

### 14. Common FIPS Compliance Pitfalls

1. **Hard-coded keys** — module loads a key from disk at startup. Not FIPS-acceptable unless the file is itself in an HSM.
2. **Non-approved RNG** — `Math.random()` or Mersenne Twister in a code path. Disallowed; use the FIPS DRBG.
3. **Mixed providers** — FIPS provider and default provider both in the same application; one call slips through.
4. **No self-test on power-up** — usually enforced by the library, but a misconfigured module can skip it.
5. **No zeroization on tamper** — the FIPS boundary test checks for this.
6. **Stale validation certificate** — vendor has *invalidated* the module by a silent update.
7. **Out-of-scope algorithms** — calling a non-FIPS algorithm (e.g., ChaCha20 in a FIPS-only module) is a violation.

### 15. The FIPS Validation Workflow for a New Module (Vendor View)

1. **Spec the module**: define the boundary, services, CSPs, roles.
2. **Write the Security Policy** (a public document).
3. **Submit to a CCTL** (e.g., atsec, ICSA Labs, Leidos, Booz Allen Hamilton).
4. **CAVP testing** of each algorithm.
5. **Module-level testing** (DTR).
6. **CMVP submission** with all artifacts.
7. **CMVP review** (6–18 months).
8. **Validation certificate** issued; module listed.
9. **Maintenance**: vendor submits changes; CMVP issues update.

The vendor *must not* claim "FIPS 140-3" until the validation certificate is issued; "in process" or "FIPS 140-3 *Ready*" is not a FIPS-validated module.

## CISO / Risk Manager View

**Board framing**: "Are we FIPS-validated for the keys that matter?" The answer is: it depends on the *use* of the key. Data-encrypting keys for application data may live in software FIPS-validated modules. CA keys, code-signing keys, root keys, *and* the master key for any key-wrapping layer must be in a Level 3 HSM.

**FIPS compliance program (CISO's checklist)**:
1. **Inventory** every cryptographic module in the estate (libraries, HSMs, KMS, cloud services).
2. **Map each module to a FIPS level** (1 / 2 / 3 / 4) and a current validation certificate (not expired).
3. **Map each key to the module that holds it**. CA keys = Level 3 HSM. Code-signing keys = Level 3 HSM. PII data keys = Level 2+ (or Level 1 in a software-validated library for non-PCI/non-HIPAA use).
4. **Verify FIPS-mode configuration** of every application; disable non-approved algorithms; switch to the FIPS provider.
5. **Annual self-test** of every FIPS module's configuration; verify POST is running.
6. **Vendor monitoring** — track the CMVP certificate status; if a vendor's certificate is *withdrawn*, plan the replacement.
7. **PQC migration plan** — every FIPS-validated module will need re-validation for ML-KEM, ML-DSA, SLH-DSA; engage vendors now.

**The FIPS boundary is the audit boundary**: if the auditor finds an *un-FIPS-validated* crypto call in a *FIPS-validated* module's path, the entire module is non-compliant. Common audit finding: a developer used a non-validated algorithm "for testing" and it survived into production.

**Common FIPS-mandated use cases (regulatory)**:
- **FedRAMP** (US federal cloud): FIPS 140-3 validated crypto is mandatory at *Moderate* and *High* baselines.
- **PCI DSS 4.0** (since 2024): strong cryptography; FIPS 140-3 is *one* way to demonstrate, but not the only way.
- **HIPAA Security Rule**: encryption "consistent with NIST standards" — FIPS 140-3 is the canonical reference.
- **CMMC** (US defense): FIPS 140-3 validated modules required for CUI.
- **FISMA** (US federal agencies): FIPS 140-3 mandatory.

**Single most important CISO callout**: the *operational* FIPS posture is what the auditor tests, not the *contractual* FIPS claim. A "FIPS-validated" HSM that is misconfigured, or a FIPS-validated software library that has been *patched past its validation* (i.e., running a binary the vendor did not submit to CMVP), is *not* FIPS-validated. The auditor's test is a copy of the *exact* binary listed in the certificate.

**The "FIPS-ready" anti-pattern**: vendors marketing "FIPS-ready" or "FIPS-compliant" before validation is issued. The CISSP must distinguish "validated" (CMVP certificate exists) from "in process" (vendor is testing) from "FIPS-compliant" (vendor claims their product is built to FIPS standards) — only the first is regulatorily meaningful.

**Board-level callout**:
- *HSM procurement* is a *multi-year* decision. Replacing an HSM in 2026 with a PQC-capable one in 2029 is a 3-year migration; planning should start now.
- *FIPS 140-4* (next iteration) is on NIST's 2025–2027 roadmap; expect new requirements on PQC and side-channel resistance.
- *FIPS validation costs*: $50k–$500k per module per algorithm set; large HSMs validate against a *set* and amortize the cost; small vendors often validate a single algorithm set.

**Board question to prepare for**: "What is our FIPS compliance score by key class? When was the last CMVP certificate audit? What is our PQC-HSM migration timeline?" The CISO's table is the answer.
